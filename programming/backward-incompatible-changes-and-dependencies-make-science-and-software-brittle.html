Backward-incompatible changes and dependencies make science and software brittle

under scipypythonscientific computingsoftware engineeringTweet

Here’s a small anecdote that reveals why the combination of backward-incompatible changes and dependencies make science and software brittle.

Scipy 0.18.0 was released 10 days ago; 6 months of hard work and awesome new features. But it changed a minor corner case in ndimage.affine_transform and this broke nilearn, our neuroimaging library, in a fairly subtle way. We detected it via indirect consequences on the continuous integration of our documentation, and I lost a day understand and fixing the problem.

However, what’s most problematic is that many others will not have the tools to audit there code and find how and why it is broken. In other words, there will be some science done with broken code, and code of published paper will stop reproducing them. In effect, this create code rot. Active software maintenance is the solution to code rot, but we cannot expect that for all published results.

Pining versions of libraries is a solution to reproduce a given scientific result, but I hate it as amount to deep freezing science: in my eyes, reusability matters as much as reproducibility for science. For libraries like nilearn, imposing strong constraints on dependency version defeats their purpose: combining these libraries becomes harder and sometimes impossible.

Dependencies enable you to use today the hard work of others, but makes you fragile to what they may change. Whether to depend on an external package for a functionality or to reimplement that functionality boils down to how hard to code this functionality.

The complex tangle of dependencies and software changes makes it very hard to guarantee that a code will give the same answers in the future.

Note

  • Use dependencies sparingly: they make us powerful, yet and fragile.
  • Refrain from backward-incompatible changes: they break other people’s code, and tarnish the trust in your code. Avoid
